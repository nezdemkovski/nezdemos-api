# Nezdemos API

Read-only API for personal data surfaces used by agents. The first module exposes
synced WHOOP data from the `whoop-scraper` PostgreSQL schema.

## Stack

- Go 1.26
- Huma v2 for typed handlers and OpenAPI generation
- chi router
- pgx pool

## Project Layout

```text
cmd/nezdemos-api/              process entrypoint only
internal/app/                  dependency wiring and API registration
internal/config/               environment config
internal/platform/postgres/    shared Postgres pool setup
internal/domain/whoop/         WHOOP domain models and use-cases
internal/adapters/postgres/    SQL repositories for domain ports
internal/httpapi/              HTTP middleware and shared routes
internal/httpapi/whoop/        WHOOP HTTP transport and Huma DTOs
```

Keep SQL in adapters, endpoint request/response glue in `httpapi`, and
aggregation/context logic in `domain`. Domain packages should not import Huma,
chi, pgx, or other transport/storage dependencies.

Keep tests in `tests/` subpackages when a package is likely to grow. Prefer
testing through public package behavior with fakes instead of exporting symbols
only for tests.

## Configuration

```bash
export DATABASE_URL='postgres://health:password@localhost:5432/health'
export NEZDEMOS_ADDR=':8080'
export NEZDEMOS_API_KEY='optional-shared-secret'
```

If `DATABASE_URL` is not set, the API reads `POSTGRES_HOST`, `POSTGRES_PORT`,
`POSTGRES_DB`, `POSTGRES_USER`, and `POSTGRES_PASSWORD`.

## Run

```bash
go run ./cmd/nezdemos-api
```

## Helm Chart

Release tags publish:

- image: `ghcr.io/nezdemkovski/nezdemos-api:<version>`
- chart: `ghcr.io/nezdemkovski/charts/nezdemos-api:<version>`

Argo CD example:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: nezdemos-api
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: homelab
  source:
    repoURL: ghcr.io/nezdemkovski/charts
    chart: nezdemos-api
    targetRevision: 0.1.0
    helm:
      valuesObject:
        image:
          tag: "0.1.0"
        env:
          NEZDEMOS_ADDR: ":8080"
          POSTGRES_HOST: whoop-postgres-rw
          POSTGRES_DB: whoop
          POSTGRES_USER: whoop
        envFrom:
          - secretRef:
              name: nezdemos-api-env
  destination:
    server: https://kubernetes.default.svc
    namespace: nezdemos-api
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

## Endpoints

- `GET /healthz`
- `GET /openapi.json`
- `GET /whoop/profile`
- `GET /whoop/latest`
- `GET /whoop/days?from=2026-05-01&to=2026-05-19`
- `GET /whoop/context?days=14`

`/whoop/context` is the main agent endpoint: it returns profile, latest day,
recent daily metrics, trend summary, interpretation hints, and data freshness in
one compact response.

The OpenAPI document is generated by Huma from typed request/response structs.
Each operation has a stable `operationId`, tags, summaries, descriptions, and
field-level schema docs so agents can use `/openapi.json` as a tool contract.

When `NEZDEMOS_API_KEY` is set, pass either:

```text
X-API-Key: <key>
Authorization: Bearer <key>
```
